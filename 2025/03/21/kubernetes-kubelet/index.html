Kubelet

kubelet 是 Kubernetes 节点（Node）上的核心组件，负责管理节点上的 Pod 和容器生命周期，是 Kubernetes 控制平面（Control Plane）与节点层之间的桥梁。

1. Kubelet API

Kubelet 提供了一系列 HTTP API 服务，用于与外部或 Kubernetes 控制平面通信。关键的端口与服务如下：

  • :10250 API - 主要用于与 Kubernetes 控制组件（如 API Server）进行双向通信，支持 gRPC 和 HTTPS 连接，具有全功能权限。
  • :10255 只读 API - 提供一个无需认证的只读接口，通常用于外部系统（如监控系统）获取 Pod 和节点的状态信息。（此端口已逐渐不推荐使用）
  • :10248 /healthz - 用于 Kubelet 自身的健康检查。主要提供 /healthz 接口，以供其他组件监控 Kubelet 运行状况。

这些接口是外界与 Kubelet 交互的重要入口。

2. 核心组件

Kubelet 的核心功能模块分工明确，各组件协同工作以完成节点管理任务。关键模块如下：

(1) syncLoop

syncLoop 是 Kubelet 的核心主循环，负责持续同步节点的状态和调度的 Pod 期望状态。这是一个事件驱动循环，具体包括以下任务:

  • 从 Kubernetes API Server 获取调度到该节点的 Pod 信息（期望状态）。
  • 对比节点上实际运行的 Pod 状态（实际状态）。
  • 调用其他模块（如 PodWorker）完成实际状态与期望状态的对齐。

(2) PodWorker

PodWorker 是每个 Pod 的具体操作执行模块。它是一个并发协程池，用于处理 Pod 的创建、更新、删除等任务。与 syncLoop 配合，PodWorker 实现了节点状态与调度期望状态的最终一致性。

(3) StatusManager

StatusManager 负责汇报节点和 Pod 的状态信息。具体工作包括：

  • 将本地运行中 Pod 的状态（如 Ready、运行时间、网络状态）上传到 API Server。
  • 将状态数据同步到 syncLoop，用于进一步决策。

(4) ProbeManager

ProbeManager 负责 Pod 的健康检查任务。它包括：

  • Liveness Probe：判断容器是否健康，不健康时触发重启。
  • Readiness Probe：判断容器是否准备就绪，用于流量负载分发。
  • Startup Probe：当容器启动较慢时，避免被误判为不健康。

ProbeManager 会根据 Pod 的探针配置，周期性调用相关检查逻辑，确保应用运行状态。

(5) cAdvisor

cAdvisor 是 Google 开源的资源监控工具，被内置为 Kubelet 的一部分，负责收集节点和容器的资源消耗（CPU、内存、网络、磁盘 IO 等）。这些数据被用于：

  • 提供给 Kubernetes 控制平面的资源调度。
  • 在节点上触发资源回收或补救机制。

(6) EvictionManager

EvictionManager 负责节点资源压力下的 Pod 驱逐。它监控节点整体资源状况，当内存、磁盘、inode 等资源低于预设阈值时，会主动驱逐“非关键”Pod 以释放资源。

(7) DiskSpaceManager & Image GC

  • DiskSpaceManager 负责监控节点磁盘空间，并协作执行相应的垃圾回收。
  • Image GC 用于定期回收未使用的镜像资源，避免磁盘因持久化镜像而耗尽。

(8) VolumeManager

VolumeManager 负责处理 Pod 的存储卷挂载和卸载，例如支持 PersistentVolume (PV) 的分配与管理。在 Pod 创建过程中，确保所需存储卷按请求挂载到容器。

(9) CertificateManager

CertificateManager 负责管理 Kubelet 所需的证书和密钥文件，用于与 API Server 安全通信的 TLS 握手。

(10) GPUManager

GPUManager 专为需要显卡支持的容器（如深度学习模型或 GPU 加速的应用）服务，确保请求 GPU 资源的 Pod 能够成功调度并正常初始化设备。

(11) OOMWatcher

OOMWatcher 监听容器是否因内存不足（OOM）被系统杀死，并触发回调通知其他模块进行后续处理。

3. 容器运行时接口（CRI）

CRI (Container Runtime Interface) 是一个抽象层，使 Kubelet 能与不同容器运行时通信。其主要组件包括：

  • CRI gRPC Server: 提供符合 Kubernetes CRI 协议的 gRPC 服务接口，是 Kubelet 和容器运行时间的桥梁。
  • Container Network Plugin: 负责调用 CNI（容器网络接口）插件配置容器的网络。Kubelet 不直接管理网络，而是通过 CNI 的规范完成网络分配。
  • Docker shim: 是早期 Kubernetes 支持 Docker 的实现。Docker shim 提供了一个中间接口，将 Docker 运行时适配到 CRI。Kubernetes 1.20 以后已逐步弃用，建议直接使用 containerd。

4. 容器运行时

在 CRI 的抽象之下，具体运行时实现通常有以下几种：

  • docker: 即经典的 Docker 容器运行时。
  • containerd: Docker 的核心组件之一，后来成为独立的容器运行时。
  • CRI-O: 专为 Kubernetes 设计的轻量级容器运行时。
  • 其他 CRI 实现: 如 Mirantis 的 cri-containerd 等。

这些运行时负责实际容器的生命周期管理，包括创建、运行、终止、删除等操作。

总结模块间协作流程

  1. 调度 Pod：API Server 通知 Kubelet 某个 Pod 被调度到当前节点。
  2. 状态同步：syncLoop 启动同步逻辑，对应 Pod 任务会通过 PodWorker 执行。
  3. 容器操作：PodWorker 调用 CRI 接口，通过容器运行时完成 Pod 的准备、启动。
  4. 存储和网络：存储卷挂载由 VolumeManager 负责，网络分配由 CNI 插件完成。
  5. 监控和健康检查：Kubelet 使用 cAdvisor，ProbeManager 收集容器和节点的运行状态，必要时执行重启或驱逐。
  6. 资源管理：通过 EvictionManager 和垃圾回收模块维持节点资源的健康。

这一架构反映了 Kubelet 在 Kubernetes 中作为“节点代理”的角色，它不仅需要协作完成 API Server 下发的期望状态，还需实时管理节点和容器状态，做到高效和可靠。
