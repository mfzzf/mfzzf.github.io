深入理解 etcd：一个分布式键值存储系统

什么是 etcd？

etcd 是一个分布式键值存储系统，由 CoreOS 基于 Raft 共识算法开发。它是 Kubernetes 控制平面组件的核心，用于存储整个集群的关键数据。etcd 的主要特点包括：

  • 键值对存储：以分层目录形式组织数据，类似文件系统。
  • 强一致性：基于 Raft 算法，确保数据一致性。
  • 监听机制：支持对键或目录的变更进行监听。
  • 高性能：单实例支持每秒 1000 次写操作和 2000 次读操作。
  • 安全性：支持 SSL/TLS 加密和客户端证书认证。

使用场景

etcd 的典型使用场景包括：

  1. 服务注册与发现：在分布式系统中，服务可以通过 etcd 注册自身，并通过 TTL（Time-To-Live）机制维护健康状态。
  2. 分布式锁和选主：通过 Compare-And-Swap（CAS）和 Compare-And-Delete 操作实现分布式锁和主节点选举。
  3. 配置中心：集中管理配置，并通过 Watch 机制动态更新配置。
  4. 存储元数据：如 Kubernetes 的 API 对象元数据存储。

核心机制

TTL（Time-To-Live）和 CAS（Compare-And-Swap）

  1. TTL：为键设置有效期，过期后键会自动删除。这在分布式锁中非常有用。
  2. CAS：对键赋值时需要满足特定条件（如键是否存在、键的当前值等），以确保操作的原子性。

Watch 机制

etcd 提供了 Watch 功能，用于监听键或目录的变更。一旦发生变更，etcd 会通知订阅者。

WAL 日志和 Raft 协议

  1. WAL 日志：etcd 使用 Write-Ahead Log（WAL）记录所有操作日志，确保数据的持久化和一致性。
  2. Raft 协议：etcd 的一致性保证基于 Raft 协议，支持选主、日志复制和故障恢复。

etcd 的存储机制详解

etcd 是一个分布式键值存储系统，其存储机制设计精巧，能够高效地支持多版本并发控制（MVCC）、事务处理以及数据压缩等功能。以下是 etcd 存储机制的详细介绍。

存储架构

etcd 的存储架构可以分为两部分：

  1. 内存中的索引（kvindex）：

    • kvindex 是基于 Google 开源的 Golang BTree 实现，用于加速键值对的查询[^1]。
    • 它保存了键（key）与版本号（reversion）之间的映射关系。

  2. 后端存储（Backend）：

    • etcd 使用 BoltDB 作为后端存储。BoltDB 是一个支持事务的嵌入式键值数据库[^1]。
    • 在 BoltDB 中，etcd 以版本号（reversion）作为键，存储键值对的完整历史版本。

多版本存储机制（MVCC）

etcd 的多版本存储机制（MVCC）允许用户访问键的历史版本，同时支持事务操作。以下是其实现细节：

版本号（Reversion）

每个键值对的版本号由两部分组成：

  • Main Revision：每次事务提交时递增 1。
  • Sub Revision：同一事务内的每次操作递增 1[^1]。

碎片整理（Defragmentation）

历史版本的压缩可能导致存储空间碎片化，这些碎片虽然不可用，但仍占据磁盘空间。etcd 支持通过 defrag 命令整理碎片[^4]。

存储配额

为了防止存储空间耗尽，etcd 提供了存储配额功能。用户可以设置合理的存储配额（如 2GB），一旦超出配额，etcd 会进入只读模式[^3]。

1. 客户端 (Client)

  • 功能： 客户端是 etcd 的使用者，可以是命令行工具（如 etcdctl），也可以是应用程序中的 SDK。
  • 交互方式： 客户端通过 gRPC 协议与 etcd 集群进行通信。
  • 注意事项：
    • 客户端通常会配置多个 etcd 节点的地址，以便在某个节点不可用时自动切换到其他节点。
    • 客户端 SDK 通常会提供负载均衡和故障转移的功能。

2. Leader 节点

  • 选举： etcd 集群使用 Raft 算法选举出一个 Leader 节点。Leader 负责处理所有客户端请求，以及协调集群中的数据复制。
  • 角色：
    • Leader： 处理客户端请求，协调数据复制。
    • Follower： 复制 Leader 的日志，参与选举。
    • Candidate： 选举过程中的临时角色，用于发起选举。

3. 预检查 (Pre-checks)

Leader 节点在处理请求之前，会进行一系列预检查，以确保请求的合法性和安全性。

  • 配额检查 (Quota Check)： 检查当前集群的使用量是否超过了预设的配额限制（例如存储空间、请求速率等）。

    • 作用： 防止单个用户或应用过度消耗集群资源。
    • 实现： etcd 会跟踪每个用户的资源使用情况，并在超过配额时拒绝请求。

  • 限速检查 (Rate Limiting)： 限制客户端请求的速率，防止过多的请求压垮集群。

    • 作用： 保护集群免受 DoS 攻击或突发流量的影响。
    • 实现： etcd 使用令牌桶算法等限速算法来控制请求速率。

  • 鉴权检查 (Authentication)： 验证客户端的身份，确保只有授权用户才能访问集群。

    • 作用： 保护集群数据安全，防止未经授权的访问。
    • 实现： etcd 支持多种鉴权方式，如 TLS 证书、用户名/密码等。

  • 包大小检查 (Packet Size Check)： 检查请求的数据包大小是否超过了预设的限制。

    • 作用： 防止过大的请求导致网络拥塞或内存溢出。
    • 实现： etcd 会在接收到请求时检查数据包大小，并在超过限制时拒绝请求。

4. KV Server

  • 功能： KV Server 是 etcd 的核心模块，负责处理客户端的读写请求。
  • 读请求： 直接从 MVCC 存储中读取数据并返回给客户端。
  • 写请求：
    1. 将写请求转换为 Raft 日志条目。
    2. 通过 Raft 模块将日志条目复制到集群中的其他节点。
    3. 当日志条目被大多数节点复制成功后，将其提交到 MVCC 存储。

5. 一致性模块 (Raft)

  • 作用： Raft 是一种分布式一致性算法，用于保证 etcd 集群中数据的一致性。
  • 核心概念：
    • 日志复制： Leader 将写请求作为日志条目复制到 Follower 节点。
    • 多数派： 只有当日志条目被大多数节点复制成功后，才认为该条目已提交。
    • Leader 选举： 当 Leader 节点故障时，Follower 节点会发起选举，选出新的 Leader。
graph LR
    subgraph Raft模块
        A[Leader] --> B(复制日志)
        B --> C[Follower1]
        B --> D[Follower2]
        B --> E[FollowerN]
    end
  • 日志复制过程：
    1. Leader 接收到写请求。
    2. Leader 将写请求追加到自己的日志中。
    3. Leader 将日志条目并行地复制到 Follower 节点。
    4. Follower 节点接收到日志条目后，将其追加到自己的日志中，并向 Leader 发送确认消息。
    5. 当 Leader 收到大多数 Follower 节点的确认消息后，认为该日志条目已提交。
    6. Leader 将提交的日志条目应用到自己的状态机（MVCC 存储），并通知 Follower 节点也应用该条目。

6. WAL 日志 (Write-Ahead Log)

  • 作用： WAL 日志用于持久化存储 Raft 日志条目，以便在节点故障时进行数据恢复。
  • 特性：
    • 顺序写入： WAL 日志是顺序写入的，可以提高写入性能。
    • 持久化： WAL 日志会持久化到磁盘上，即使节点重启也不会丢失数据。
    • 故障恢复： 当节点故障重启后，etcd 可以通过重放 WAL 日志来恢复数据。

7. MVCC 存储 (Multiversion Concurrency Control)

  • 作用： MVCC 存储是 etcd 的底层存储引擎，用于存储键值对数据。
  • 特性：
    • 多版本： MVCC 存储会保存键值对的多个历史版本，可以实现快照读和历史数据查询。
    • 并发控制： MVCC 存储使用乐观锁等机制来处理并发读写请求。
    • 事务支持： MVCC 存储支持原子事务，可以保证多个操作的原子性。
